AfterglowRender

序言

开发背景

大约一年以前, 我产生了利用假期和 996 空余时间搭建一个自己的 实时渲染框架 的想法. 产生该想法的原因是, 我平常在阅读各类书籍和文献的过程中, 了解到许多精妙的图形渲染的实现和优化方法. 我层尝试将这些方法在自己的工程中复现, 但其中很部分方法需要对渲染流程深度定制. 当时苦于对完整渲染流程认知的不完善, 一直难有机会将这些方法进行实践. 然而图形学却偏偏是一门实践性很强的学科, 仅阅读而不付诸实践, 恐怕难以把认知落实为真正的知识和经验.

出于对深入了解图形渲染和 GPU 通用计算方法底层实现的愿景, 我萌生了基于底层图形 API 设计和搭建一套基于 光栅化渲染 的可交互程序框架的想法.

其实单从产出作品的角度来说, 藉由现代游戏引擎, 如 Unreal 或 Unity 要远比从图形 API 底层开始实现效果更好, 也更高效. 不过正是由于现代游戏引擎过于强大, 强大到几乎掩盖了所有恼人的运作细节和设计原理, 致使在使用游戏引擎的过程中, 常常陷入到 "知其然而不知其所以然" 的状态. 始终难以更加深层地建立对功能背后技术原理的认知., 身陷于对的技术特征用法层面的疲乏追赶; 在不了解技术特征背后机理的情况下, 也容易在现有对技术特征产生误用, 产生额外的排错和调试成本.

因此, 我将以 VulkanTutorial 为起点[1], 目标逐步搭建一套集实时渲染, 模拟及交互的图形程序框架; 作为深入了解图形底层原理, 构建完善的知识体系认知的桥梁.

设计定位

在权衡考量开发时间, 预期收益和技术难度等因素后, 我决定将该程序的核心目标用途定义为以下两点:

  • 验证平台: 该程序将作为便捷验证 Shader 效果, 图形技术实现, 以及 玩法交互设想 的基础框架. 为达成该目的, 程序框架设计应当尽量通用, 提供充分的扩展空间, 并对基础图形接口进行封装, 以简化功能扩展带来的非必要理解负担. 此外, 对于一些需要频繁反复调试的行为, 例如测试 Shader 效果, 调控组件交互行为等, 应当提供 运行时 编辑方案, 而不依赖于对源码的修改.

  • 辅助工具: 随着程序框架搭建的逐步完善, 其将借助自身现有功能为常规工作提供辅助, 比如基于通用 GPU 计算能力提供程序化的贴图或模型等资产的 批量生成; 亦或是对已有资产进行 再加工; 再或是对非通用格式的顶点或贴图数据进行 解析和转换 等.

Entity-Components 模式

在历经 VulkanTutorial 千余行代码绘制的第一个三角形后, 如何把一个三角形变成几个三角形, 是又一个严峻的挑战. 该问题更加本质地讲, 是需要去构建一套组织框架, 建立对象之间的关联模型, 管理各类资源并与底层图形 API 对接, 并需要提供一套易于增删查改的接口范式.

不夸张地说, 以 Vulkan API 的稀碎程度, 如果强行在不进行任何封装的情况下尝试直接添加绘制对象, 每一次新增的绘制恐怕都需要在整个工程的十余处位置进行改动, 十分考验心智.

为此, 我采用了现代游戏引擎中广泛使用的组织模式, 以 场景 (Scene) 作为对象容器, 实体 (Entity) 作为基本单元, 通过在实体中组合不同的 组件 (Component), 即可在场景中实现出各式各样的效果.

材质系统

说到材质, 接下来我要隆重介绍 AfterglowRender 的另一项重要特性: 材质系统.

材质系统实质上是对 Vulkan 底层 DescriptorSet, DescriptorSetLayout 相关的一大堆东西在 C++ 和 Shader 两端的封装, 其中关联的细节较多, 会在后续的 技术细节 中详细提及.

当前主要展示材质系统带来了哪些好处:

  • 自动生成 Shader 绑定信息: 无需因 C++ 内的声明改动而来回修改 Shader 代码声明.
  • 简化资源和参数配置: 通过简单的操作即可为 Shader 载入资产和参数, 而无需关心背后复杂的资产管理和描述符提交.
  • 运行时 Shader 编译和渲染管线重建: 你可以在程序运行时对 Shader 代码和 材质参数资产进行修改, 改动可以实时作用到画面上, 无需再次运行程序或重新编译.
  • 供逻辑组件调用: 通过材质系统接口, 也可以从诸如 ActionCompnent 的逻辑代码对材质进行修改和增删.

ComputeShader 通用计算

说完材质系统的演示内容, 接下来我想分享一下 AfterglowRender 所支持的另一项重要技术, ComputePipeline, 这也是我开发这个程序框架想要验证的重点内容之一.

提起 ComputeShader, 最容易联想到的是 Unity 为开发者提供的 ComputeShader 工作流[9], 借助 ComputeShader 可以实现高效的并行计算, 也可以实现五花八门的表现效果. 除去这类的自定义拓展外, ComputeShader 实际上已经渗入了当今实时渲染流程中的每一处隙缝. 例如 Unity 的 VFX; Unreal 的 Niagara, Lumen, Nanite; 或是 雾锁王国 (Enshrouded), 无人深空 (No Man's Sky) 等游戏的体素化地形, 其中都能看到 Compute Shader 的影子.

在搭建 Compute Pipeline 流程时, 我比较纠结将其放在什么位置合适, 它虽然是独立于渲染流程的内容, 但是在 Vulkan 描述构造和提交上却又有相当多的共同点. 最终, 我决定将其作为材质系统的一个 扩展. 一方面原因是诸多现有方法都可以复用; 更重要的因素在于, 它最终可以具备如上述 材质系统 般的运行时编辑和及时预览功能, 这对于提升开发和验证效率有巨大帮助.

详细的流程构造会在 通用计算流程设计 中展开, 目前先介绍已经过验证的 ComputeShader 用法.

  • 初始化并实时更新顶点数据: 利用 ComputeShader 初始化并逐帧更新 StorageBuffer 中存储的顶点运动数据, 便可以得到一个简化的粒子特效生成器. 如下图展示, 生成并在运行时修改粒子数据:
通过材质系统对 GPU 粒子进行调试

未来工作

GPU-Picking 和 反射式组件功能 UI

为了进一步强化调试的便捷性, 摆脱对源码修改的依赖 (哪怕已经尽力减少编译依赖, 在频繁的改动下编译上花费的时间仍不容小觑), 我计划在未来引入 GPU-Picking 和 组件功能 UI.

通过光标直接获取Entity ID, 并绘制关联组件的功能和属性编辑页面. 在组件页面的交互功能的处理上, 由于先前已经实现了一套 静态反射 方案, 因此我计划直接通过反射构建组件页面内容, 实现直接在页面上修改对象属性或是调用成员函数的功能.

场景序列化和反序列化 Serializer

我计划构建或者引入一套组合序列化体系, 通过实现基础类型序列化方法, 辅以反射信息, 实现对复杂组合对象的序列化. 具备序列化能力后, 即可推进现有类型如 Scene, Entity 等的持久化处理, 并进一步构建出资产体系.

扩充完善渲染流程

虽然在最初设计时, 就已经预留了完整 Subpasses 的位置, 但目前仅实现了少数几个支持基本渲染功能 Subpasses. 诸如 Pre-Z, Shadow, Decal 等 Subpasses 尚待支持.

更灵活的通用计算架构及更多演示示例

Compute Shader 蕴藏着无限的可能. 除支持 ComputeTask 之间共享 SSBO 数据, CPU-GPU 端数据低频同步, 生成间接绘制命令等基础功能的扩展外, 许多依托于此的应用, 如 Boids 集群控制, 物理粒子模拟, 预计算数据生成等, 都还存在广泛的探索空间.

更多渲染特性支持

回到最初搭建这套框架的构想, 是希望能够借助其完成对不同的渲染技术进行验证, 并深入了解它们的基本运作机理. 后续基础功能搭建到位后, 便会逐步将重心转向渲染特性和效果的验证上, 如 物理镜头视效, 天空大气和体积雾, 顺序无关的半透明渲染等.

此外, 延迟渲染 也具有十足的吸引力, 引入 延迟渲染 就意味着场景对多重光源的掌控和优化能力得到显著提升, 更重要的是, 丰富的 GBuffer 信息也昭示着琳琅满目的屏幕空间算法将得以施展. 不过引入延迟渲染也意味着图形 API 自带的 MSAA 将难以施展, 如何重新构建一套合适的抗锯齿方案, 并处理于现有渲染流程的潜在兼容问题, 预计也是不小的挑战.

设计优化

虽然 AfterglowRender 完全遵从 RAII 约定, 全局没有任何 new 和 del, 但一些贯穿程序生命周期的容器依然可能造成类似内存泄漏的影响. 此外, 为避免当前的多线程架构线程冲突所添加的线程锁, 也一定程度上破坏了原先的封装性, 值得对其设计模式进行重新思考.
