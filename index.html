序言

开发背景

大约一年以前, 我产生了利用假期和 996 空余时间搭建一个自己的 实时渲染框架 的想法. 产生该想法的原因是, 我平常在阅读各类书籍和文献的过程中, 了解到许多精妙的图形渲染的实现和优化方法. 我层尝试将这些方法在自己的工程中复现, 但其中很部分方法需要对渲染流程深度定制. 当时苦于对完整渲染流程认知的不完善, 一直难有机会将这些方法进行实践. 然而图形学却偏偏是一门实践性很强的学科, 仅阅读而不付诸实践, 恐怕难以把认知落实为真正的知识和经验.

出于对深入了解图形渲染和 GPU 通用计算方法底层实现的愿景, 我萌生了基于底层图形 API 设计和搭建一套基于 光栅化渲染 的可交互程序框架的想法.

其实单从产出作品的角度来说, 藉由现代游戏引擎, 如 Unreal 或 Unity 要远比从图形 API 底层开始实现效果更好, 也更高效. 不过正是由于现代游戏引擎过于强大, 强大到几乎掩盖了所有恼人的运作细节和设计原理, 致使在使用游戏引擎的过程中, 常常陷入到 "知其然而不知其所以然" 的状态. 始终难以更加深层地建立对功能背后技术原理的认知., 身陷于对的技术特征用法层面的疲乏追赶; 在不了解技术特征背后机理的情况下, 也容易在现有对技术特征产生误用, 产生额外的排错和调试成本.

因此, 我计划以 VulkanTutorial 为起点[1], 目标是逐步搭建一套集实时渲染, 模拟及交互的图形程序框架; 作为深入了解图形底层原理, 构建完善的知识体系认知的桥梁.

设计定位

在权衡考量开发时间, 预期收益和技术难度等因素后, 我决定将该程序的核心目标用途定义为以下两点:

  • 验证平台: 该程序将作为便捷验证 Shader 效果, 图形技术实现, 以及 玩法交互设想 的基础框架. 为达成该目的, 程序框架设计应当尽量通用, 提供充分的扩展空间, 并对基础图形接口进行封装, 以简化功能扩展带来的非必要理解负担. 此外, 对于一些需要频繁反复调试的行为, 例如测试 Shader 效果, 调控组件交互行为等, 应当提供 运行时 编辑方案, 而不依赖于对源码的修改.

  • 辅助工具: 随着程序框架搭建的逐步完善, 其将借助自身现有功能为常规工作提供辅助, 比如基于通用 GPU 计算能力提供程序化的贴图或模型等资产的 批量生成; 亦或是对已有资产进行 再加工; 再或是对非通用格式的顶点或贴图数据进行 解析和转换 等.

Entity-Components 模式

在历经 VulkanTutorial 千余行代码绘制的第一个三角形后, 如何把一个三角形变成几个三角形, 是又一个严峻的挑战. 因为该问题更加本质地讲, 是需要去构建一套组织框架, 建立对象之间的关联模型, 管理各类资源并与底层图形 API 对接, 并需要提供一套易于增删查改的接口范式.

不夸张地说, 以 Vulkan API 的稀碎程度, 如果强行在不进行任何封装的情况下尝试直接添加绘制对象, 每一次新增的绘制恐怕都需要在整个工程的十余处位置进行改动, 十分考验心智.

为此, 我采用了现代游戏引擎中广泛使用的组织模式, 以 场景 (Scene) 作为对象容器, 实体 (Entity) 作为基本单元, 通过在实体中组合不同的 组件 (Component), 即可在场景中实现出各式各样的效果.

材质系统

提到材质, 接下来我想隆重介绍 AfterglowRender 的一项重要特性: 材质系统.

材质系统实质上是对 Vulkan 底层 DescriptorSet, DescriptorSetLayout 相关的一大堆东西在 C++ 和 Shader 两端的封装, 其中关联的细节较多, 会在后续的 技术细节 中详细提及.

当前主要展示材质系统带来了哪些好处:

  • 自动生成 Shader 绑定信息: 无需因 C++ 内的声明改动而来回修改 Shader 代码声明.
  • 简化资源和参数配置: 通过简单的操作即可为 Shader 载入资产和参数, 而无需关心背后复杂的资产管理和描述符提交.
  • 运行时 Shader 编译和渲染管线重建: 你可以在程序运行时对 Shader 代码和 材质参数资产进行修改, 改动可以实时作用到画面上, 无需再次运行程序或重新编译.
  • 供逻辑组件调用: 通过材质系统接口, 也可以从诸如 ActionCompnent 的逻辑代码对材质进行修改和增删.

ComputeShader 通用计算

说完材质系统的演示内容, 接下来我想分享一下 AfterglowRender 所支持的另一项重要技术, ComputePipeline, 这也是我开发这个程序框架想要验证的重点内容之一.

提起 ComputeShader, 最容易联想到的是 Unity 为开发者提供的 ComputeShader 工作流[9], 借助 ComputeShader 可以实现高效的并行计算, 也可以实现五花八门的表现效果. 除去这类的自定义拓展外, ComputeShader 实际上已经渗入了当今实时渲染流程中的每一处隙缝. 例如 Unity 的 VFX; Unreal 的 Niagara, Lumen, Nanite; 或是 雾锁王国 (Enshrouded), 无人深空 (No Man's Sky) 等游戏的体素化地形, 其中都能看到 Compute Shader 的影子.

在搭建 Compute Pipeline 流程时, 我比较纠结将其放在什么位置合适, 它虽然是独立于渲染流程的内容, 但是在 Vulkan 描述构造和提交上却又有相当多的共同点. 最终, 我决定将其作为材质系统的一个 扩展. 一方面原因是诸多现有方法可以复用; 更重要的因素在于, 如此设计可以让它最终可以具备如上述 材质系统 般的运行时编辑和实时预览功能, 这对于提升开发和验证效率有巨大帮助.

未来工作

GPU-Picking 和 反射式组件功能 UI

为了进一步强化调试的便捷性, 摆脱对源码修改的依赖 (哪怕已经尽力减少编译依赖, 在频繁的改动下编译上花费的时间仍不容小觑), 我计划在未来引入 GPU-Picking 和 组件功能 UI.

通过光标直接获取Entity ID, 并绘制关联组件的功能和属性编辑页面. 在组件页面的交互功能的处理上, 由于先前已经实现了一套 静态反射 方案, 因此我计划直接通过反射构建组件页面内容, 实现直接在页面上修改对象属性或是调用成员函数的功能.

场景序列化和反序列化 Serializer

我计划构建或者引入一套组合序列化体系, 通过实现基础类型序列化方法, 辅以反射信息, 实现对复杂组合对象的序列化. 具备序列化能力后, 即可推进对现有如 Scene, Entity 等对象的持久化处理, 并进一步构建出资产体系.

扩充完善渲染流程

虽然在最初设计时, 就已经预留了完整 Subpasses 的位置, 但目前仅实现了少数几个支持基本渲染功能 Subpasses. 诸如 Pre-Z, Shadow, Decal 等 Subpasses 尚待支持.

更灵活的通用计算架构及更多演示示例

Compute Shader 蕴藏着无限的可能. 除支持 ComputeTask 之间共享 SSBO 数据, CPU-GPU 端数据低频同步, 生成间接绘制命令等基础功能的扩展外, 许多依托于此的应用, 如 Boids 集群控制, 物理粒子模拟, 预计算数据生成等, 都还存在广泛的探索空间.

更多渲染特性支持

回到最初搭建这套框架的构想, 是希望能够借助其完成对不同的渲染技术进行验证, 并深入了解它们的基本运作机理. 后续基础功能搭建到位后, 便会逐步将重心转向渲染特性和效果的验证上, 如 物理镜头视效, 和 体积雾, 顺序无关的半透明渲染 等.

此外, 延迟渲染 在图像效果的提升方面也具有十足的吸引力, 引入 延迟渲染 意味着场景对多重光源的掌控和优化能力可以得到显著提升. 更重要的是, 丰富的 GBuffer 信息也昭示着琳琅满目的屏幕空间算法将得以施展. 不过引入延迟渲染也意味着渲染流程会与目前图形 API 自带的 MSAA 难以兼容, 如何重新构建一套合适的抗锯齿方案, 并处理于现有渲染流程的潜在兼容问题, 也将会是不小的挑战.

设计优化

容器管理: 虽然 AfterglowRender 完全遵从 RAII 约定, 全局没有任何 new 和 del, 但一些贯穿程序生命周期的容器依然可能造成类似内存泄漏的影响. 此外, 为避免当前的多线程架构线程冲突所添加的线程锁, 也一定程度上破坏了原先的封装性, 值得对它们的设计模式进行重新思考.

渲染排序: 目前绘制实例的命令提交是无序的, 按照计划后此后会补齐自动排序以及通过 StaticMeshComponent 手动设置优先级的功能.

绘制剔除: 对于模型资产的加载, 后续计划强制引入 AABB 的计算, 单实例在 Host 端实行剔除; 多实例则利用 ComputeShader 计算绘制指令, 通过 间接绘制 (DrawIndexedIndirect | DrawIndirect) 实行精确剔除.

ComputeShader 通用计算流程设计

我设计了一条以 ComputeTask 为核心, 围绕 ComputeShader 构建 StorageBuffer, 再将 StorageBuffer 用于各类用途的工作流. 不过基于目前测试用例较少的现状, 通用计算流程中存在应该还有潜在的完善和拓展空间.

着色器语言和编译器方案

Shader 语言上, 我选择了在语法扩展性和周边支持都更为完善的 HLSL (横向对比 GLSL 甚至无法原生支持 #include); 因此也顺理成章地选择了对 HLSL Vulkan 语义拓展更好 DXC 作为主要的着色器编译器.

DXC 会在运行时接收经过材质资产 (MaterialAsset) 组装的完整 ShaderCode, 并将其编译为 SPIR-V 中间语言供 GraphicsPipeline 和 ComputePipeline 使用.

C++ 编译和构建方面, 直接使用 IDE 预设方案, 因为手动折腾构建估计要花费不少时间. 虽然此前也考虑过引入反射代码信息生成的需求, 不过经过包装的反射信息注册写起来还算简单, 需要注册反射信息的类也较少, 因此先暂时搁置了此事项.

Oct 3, 2025 @note : 该章节下的内容很容易在开发过程中发生变更, 因此我计划优先编写已经定型的部分或是不依赖具体实现的技术细节.

基础 Vulkan 类型封装代理 ProxyTemplate

Vulkan 的接口基于 C 构建 [11], 其大多数时候与现代 C++ 编程范式格格不入. 虽然也有诸如 Vulkan-hpp 的封装 [12]. 但基于某些历史原因 (Vulkan 学习曲线太陡峭, 以至于我最初学习时只能先囫囵吞枣, 不完全照着教程写跑不起来), 我决定手动对 VulkanAPI 进行封装; 以此来加深对 Vulkan 的理解, 并基于软件的设计意图提供更内聚, 使用更便捷的封装形式.

我的封装思路是, 设计一个 代理模板类, 该类型提供一些规范性约束以及接口, 所有 Vulkan 类型封装都需要继承代理类型. 为方便无缝衔接原生 Vulkan 方法, 该代理类也会提供对原始 Vulkan 类型的隐式转换.

CRTP Function

从 ProxyTemplate 的声明不难看出, 它是一个 CRTP (Curiously Recurring Template Pattern) 模板类. CRTP 主要为代理类提供 initCreateInfo 和 create 两个编译期多态函数, 用于让具体的 Vulkan 代理类能在 vulkan 对象在真正需要时被正确创建和初始化.

ProxyArray

ProxyArray 则是用于批量创建 Vulkan 对象的代理类, 相比于 ProxyObject::Array (std::vector<ProxyObject>), 它可以确保 Vulkan 对象 (指针) 间的 内存连续性, 用于适配一些 Vulkan 接口所需的数组型数据.

方法名 描述
Type& operator[] (...) 通过下标获取对应索引的 Vulkan 对象.
operator const Type* () 隐式转换为Vulkan对象类型的数组起始位置地址.
uint32_t size() const 数组大小, 由构造时创建.
void destroy(...) 同 ProxyObject.

ProxyArray 的派生类实现上与 ProxyObject 类似, 但需要在基类构造时提供数组数量参数, 此处不过多赘述. 在使用方面, 也可以直接作为 vulkan 函数对象数组类型的参数传入, 或是通过下标索引某个位置的 vulkan 对象.

AsElement

由于 ProxyObject 或 ProxyArray 的派生类可能会缺少默认构造方法, 在组合到更上的级类中或是作为容器元素中时会造成一定的不便. 因此, 代理类提供了 Proxy::AsElement 类型. AsElement 本质上是一个智能指针, 并提供代理类中大部分方法的直接调用.

由于 AsElement 不会直接构建 Proxy 对象, 而是通过 Proxy 类的 makeElement 静态函数或 AsElement 实例的 recreate 函数构建, 因此它可以很轻易的整合到其他类或容器中.

编译期类型与运行时 TypeIndex 互相转换的魔法

"能在运行时获取实例类型并进行对应处理" 是相当有吸引力的特性, 在现代 C++ 的标准库中已经存在了一些基于 类型擦除 实现的多类型容器, 如 std::any, std::variant 等[13, 14]. 但这些容器在获取元素时需要开发者显式指定元素的编译期类型. 但如果我们不知道元素的具体类型, 仅持有对象的运行时类型标识 std::type_index, 又应当如何处理?

众所周知, C++ 至今尚未支持反射特性, 虽然相关提案一直存在[15], 但在短期内还看不到其真正被支持和普及的迹象.

好在我们可以基于现有的模板元工具构建一套从运行时信息反推编译期类型的体系. 缺点是想要达成此目标, 需要让代码满足一定的设计规范.

静态反射 Inreflect

在开发过程中, 我愈发察觉到构建一套 静态反射 方案的重要性. 从获取枚举量名称, 遍历枚举值; 到根据运行时字符串查询变量, 获取变量名称; 又或是根据条件查询类中符合条件的函数名称和调用等. 都离不开反射的支持.

在没有反射以前, 通常只能根据上述的某一个特定需求, 对类型做出特殊的修改和规范定制, 长此以往不但容易造成功能的重复定义, 并降低类的可读性; 不同时间在不同类型中各自的定义的方法也极易产生差异, 造成应用层使用的割裂. 在支持反射机制后, 仅需通过对需反射类型进行注册, 即可同时满足上面提及的所有需求, 并做到接口形式的统一易用.

实现原理

Inreflect 基本属于非侵入式静态反射方案, 反射声明可以与类型声明分离 (除需要获取私有成员反射信息的情况). 编译期字符串的设计参考了 UsRefl 的实现[16].

Inreflect<Class> 类型的实质是特化的模板类实例, 遵循注册信息提供各类编译期反射功能.

Inreflect 声明形式 InreflectDifinition 基类提供通用方法, 因为特化类型定义会覆盖非特化定义

Inreflect 支持多继承类型, 多态类型以及枚举类, 但是菱形继承尚未进行过测试.

支持 Inreflect 运作的核心机制是其中的 forEach 函数, 它们能够在编译期遍历所有的类成员以及所有基类成员的元信息, 并对外提供回调机制.

forEachAttribute 实现

虽然该函数的模板参数类型看起来张牙舞爪, 但其多数为内部递归特化所需参数. 在实际使用时, 我们完全不需要手动指定哪怕其中任何一个模板参数.

反射类注册

解释完 Inreflect 的基本原理, 接下来只需要对类型注册反射信息进行注册, 即可享受到反射元信息带来的诸多便利.

对反射信息的注册, 实质上就是对 Inreflect<Class> 特化类型的声明. 如果不经任何包装手写反射信息注册, 需要有一定的理解门槛. 考虑到接下来一段时间内可能都不会引入自动注册机制, 我定义了一套注册宏, 大幅简化了注册反射的理解门槛和所需编写的信息量. 在没有自动注册机制的情况下, 这些注册宏应该算是编写反射信息注册的最简形式了.

注册宏定义 注册反射信息演示
